01. AI Agent 理论基础

重新定义大模型：从“知识问答框”到“任务执行器”

SYSTEM_DEFINE

什么是 AI Agent？

核心定义

AI Agent 是一种能够感知环境、进行自主规划、调用外部工具并实现既定目标的智能系统。它将大模型（LLM）从单纯的文本生成器升级为了系统的大脑控制器。

计算演进公式

Agent = LLM + Planning +
Memory + Tools

基于强化学习与目标驱动（Goal-oriented），Agent 能够自主处理执行过程中的不确定性，遇到错误主动反思并寻找新路径，实现真正的“数字员工”闭环。

RAG 演进：检索增强的升维之路

v1

Naive RAG

关键词/向量检索 + 生成。依赖余弦相似度匹配，受限于语义鸿沟。

v2

Advanced RAG

增加重排 (Rerank)、查询改写、分块优化。大幅提升命中率与精准度。

v3

Graph RAG

融合知识图谱（实体/时间四元组），支持跨文档的深度多跳关联推理。

v4

Agentic RAG

智能体驱动，具备意图路由、主动补全、多次交互与自我纠错验证能力。

MODULE 01: PERCEPTION LAYER

感知层设计：智能体的“五感”解析器

感知层是防止“垃圾进、垃圾出”的第一道防线。核心在于将复杂现实世界的异构数据、冗余口语转化为大模型可理解的高质量结构化 Payload。

多源异构数据采集
PDF / DB / API / 语音
预处理与清洗
OCR / Chunking / 过滤噪音
意图识别与槽位提取
轻量模型序列分类

预处理与结构化切片

对密集型财报、表格进行智能切片（Chunking），剥离页眉页脚等干扰项，转化为标准 JSON。解决传统按长度暴切导致的语义截断痛点。

意图与槽位精准抽取

最新实践：采用微调后的 1.5B 轻量化模型 + Embedding 进行极速意图分类与关键参数（Slot）提取，毫秒级响应，大幅降低 Token 成本。

MODULE 02: COGNITION LAYER (BRAIN)

决策层设计：构建可拆解的复杂逻辑链

目标管理接收
校验/反问补齐参数
思维链 (CoT) 推理
模拟解题步骤拆解
计划与调度编排
生成资源与工具调用树

目标管理 (Goal)

防止系统失控。老板想查询报税数据时，决策层不知道具体企业，会向老板追问具体企业、具体账期等核心信息后继续执行。

推理与规划 (Plan)

面对复杂业务分析，系统能自主拆分为：1.查库获取维度 -> 2.执行SQL汇总 -> 3.渲染饼图 等原子级任务的调度序列。

记忆管理系统

融合瞬时上下文(Context)、任务状态短期记忆(State)以及沉淀成功经验的长期记忆(RAG图谱)，辅助下一步规划。

MODULE 03: ACTION LAYER

执行层设计：突破次元壁的物理手脚

执行层是 Agent 与外部环境（API、SaaS系统、物理世界）交互的“手和脚”。设计重点在于高可用性、安全隔离以及协议的标准化。

Tool Calling

统一 API 调用协议。根据大模型输出的 JSON 参数，自动触发企业内部 OA、财税系统能力。

Code Interpreter

在隔离沙盒中运行 Python/SQL。大模型仅生成代码并交由执行层运算。

MCP 协议基建

通过MCP（Model Context Protocol），实现外部能力的即插即用，打通系统之间的数据孤岛。

MODULE 04: REFLECTION LAYER

反思层设计：智能体的自我进化机制

反思层（Reflection）是 Agent 区别于传统自动化脚本的核心。它不盲目接受 LLM 或检索系统的初步结果，而是引入“批判者”角色进行二次校验与闭环纠错。

核心反思机制

  • 自我批评 (Self-Criticism)： 检查生成的执行计划是否逻辑自洽，是否遗漏了用户设定的关键约束条件。
  • 结果验证 (Verification)： 检查生成的代码是否报错、SQL 查询结果是否为空、调用的外部 API 是否返回失败。
  • 幻觉拦截： 通过严格对比检索到的事实源文档，确保最终输出结果非虚构。

典型应用：ReAct 与自愈

Thought: 尝试查询 A 公司财报。
Action: get_finance_data(company="A")
Observation: API 报错，缺失必填参数 year。
Reflection: API 缺少年份，我需要重试。
Thought: 重新组装参数，查询 2026 年数据。

03. 核心机制：记忆与协作

让智能体“越用越聪明”，从单兵作战走向团队协同

Memory：打破单轮对话的“遗忘”魔咒

1

瞬时记忆

机制：将当前 Session 原始的交互数据（如用户指令、SQL报错）暂存在 Redis 中。

功能：支撑当前的上下文意图补全，当超出阈值时会自动触发遗忘清理，避免污染。

2

短期记忆

机制：作为智能体的“临时工作台”，存储与当前核心任务强相关的信息快照。

功能：用于复杂多步任务（如连续三个接口测试）的状态维系，任务完结即消融。

3

长期记忆

机制：经过价值筛选（提炼），将高频特征、用户偏好持久化写入向量/图数据库。

功能：经验的宝库。遇到新 Bug 时自动复用历史成功代码模板，实现跨域泛化。

多智能体编排模式

Multi-Agent：模式A - 层级式 (Hierarchy)

架构：项目经理制

存在一个全局把控的“Planner”主节点，负责接收大目标并垂直拆解；下属多个具备特定 Profile 的“Worker”智能体各自领受子任务并行处理，最后由主节点汇总渲染。

  • 优点： 权责清晰，顶层控制力强，结果易收敛。
  • 场景： 复杂的跨表财报分析、年度经营报告自动生成。
Manager (Planner)

Multi-Agent：模式B - 流水线 (Pipeline)

Agent 1: 需求解析
Agent 2: 代码/SQL 生成
Agent 3: 渲染图表可视化

架构：工厂装配线

任务在智能体之间单向传递流转。上游 Agent 提取的结构化特征直接作为下游 Agent 的 Prompt 组装变量。如同软件工程的工作流（Workflow）。

  • 优点： 流程高度标准化，资源消耗可预估，执行效率极高。
  • 场景： 销售趋势报表固定生成、标准化代码测试覆盖。

04. 落地实战与最佳实践

直击生产环境的架构经验与系统演进

架构对比：Workflow vs Agentic RAG

智能客服个性化商品推荐架构

真实案例解析：智能客服个性化商品推荐

业务痛点： 传统客服极度依赖人工经验，面对 B 端客户无法秒级判断企业税务状态，导致推荐的产品包转化率极低。

1. 画像动态嗅探

Agent 主动调用后台 API，核实客户最新资质为“小规模纳税人”，且查询历史订单发现“未购买基础记账产品”。

2. 策略动态推理

基于知识图谱匹配：策略 Agent 推理客户首要痛点为基础合规，主动拦截高级税务筹划产品的推荐逻辑。

3. 精准推荐输出

自动组合“基础版记账 SaaS”并附加极简报税图谱生成最终话术。实现千人千面，转化率提升 40%。

为什么必须转向 Agentic 架构？

深度个性化推荐

打破死板的“If-Else”硬编码规则树。通过感知层动态拉取用户的画像标签与历史会话特征，由大模型实时推演组合最适配的服务体验。

动态决策与自愈

面对包含多维变量和缺失信息的复杂诉求，系统能主动发起 Re-asking（反问）补齐参数，并在 API 异常时自主寻优切换备选策略。

可持续学习进化

沉淀业务的数字资产。通过 Memory 系统记录研发 Bug 处置过程或金牌销售的成功话术，化个人经验为系统的全域长期记忆库。

最佳实践：模块化设计与可配置化

极度解耦的架构

绝不将业务逻辑写死在大模型 Prompt 里。

方法： 采用微服务架构，将业务风控规则、系统状态流转抽出为独立的服务节点，与大模型的规划推理彻底物理隔离。

好处： 大幅提升系统可维护性与确定性，避免大模型“幻觉”污染核心业务数据，同时实现业务逻辑的热更新而无需重新调整复杂的 Prompt 框架。

提示词与预设可配置

将智能体的 Profile（如角色设定、严格的 JSON 格式要求、特定话术红线）提取为配置中心文件。让业务人员能在可视化面板进行快速微调，降低算法研发对硬代码的依赖。

最佳实践：HITL (人机协同机制)

破除全自动化“黑盒”迷信

在当前技术水位下，对重要 ToB 场景推行 100% 端到端自治是极其危险的。

必须在系统中巧妙设计 Human-in-the-loop (环内人类) 机制。将人类定位为“最终决策辅助者”与“审批把关人”。

典型介入场景设计

  • AI 记账报税： 遇到自动化记账报税失败或账目逻辑冲突，系统挂起并自动交由人工介入处理。
  • AI 开票服务： 在开票信息识别异常、客户情绪出现异常（通过文本情感感知）、或开票过程异常中断时，强制人工客服无缝接入。

最佳实践：原子能力与 Skills 封装

封装原子能力 (Skills)

将复杂的业务操作打散为独立的、可复用的原子级技能 (Skills)，并在此基础上进行高阶抽象与封装。

1. 方法论层次封装

将企业内部专家的行业经验（如：小规模纳税人报税SOP）固化为标准化的 Prompt 模板或工作流链条。

2. 决策逻辑封装

将业务风控规则、条件判断逻辑封装为独立的判断类 Skill，由大模型在规划阶段调用，而非自行计算。

3. 行为模式封装

将系统 API（如：生成开票链接、查询 CRM）封装为标准化工具包，统一输入输出格式，供模型随时按需组合调用。

05. 技术架构的演进前瞻

向自主系统与标准化协作协议跃迁

演进一：单任务工具 → 全景自主系统

Current：被动触发单节点

目前的智能体多数仍需要用户的初始 Prompt 唤醒，执行完毕单次数据分析、报告生成或信息梳理后即陷入休眠。

Future：自主运行 (Autonomic)

基于底层的长效 Memory 与定时唤醒机制。系统将具备后台长驻能力，自主进行全网态势感知、服务器监控巡检，当检测到阈值偏离时，无需指令主动生成并下发修复方案。

演进二：单一 RAG → Agentic RAG

检索增强系统自身将演化为一个包含内部自我管理逻辑的“活体图谱”，而非机械的向量数据库。

数据自清洁

系统能够在与大量用户交互的过程中，主动发现知识库文档中陈旧或冲突的规章制度，并在后台标记打分。

隐式多跳重构

不再单纯依靠抽取四元组，系统在运行时能动态抽象不同知识片断的潜在逻辑，形成具备自适应权重的神经检索网络。

知识主动反哺

当系统发现连续多位用户提问某新领域且检索失败时，触发预警，自动生成需求清单，反向提醒业务人员补充专门的预料库。

演进三：私有接口 → Skills & MCP 协议基建化

模型上下文协议 (MCP) 统一

过去，每增加一个外部能力（调日历、读财务DB）都需要写大量粘合代码，导致大模型工具链极度碎片化和高耦合。

未来，MCP（Model Context Protocol） 将成为大模型时代的“USB接口标准”。任何企业资产只需暴露为标准 MCP 服务，底层不同厂商的 AI 就能像装载插拔件（Skills）一样，零代码自动解析并无缝调用，打通数据孤岛。
